# 1️⃣ 학습 내용 정리

## 1. 스토어드 프로시저 사용 방법 

### 1-1. 스토어드 프로시저의 개념
* **정의:** MySQL에서 제공하는 **프로그래밍 기능**으로, 쿼리 문의 집합으로 볼 수 있습니다.
* **목적:** 자주 사용하는 일반적인 쿼리를 매번 반복해서 작성하는 대신, 스토어드 프로시저로 묶어 놓고 필요할 때마다 간단히 호출(`CALL`)하여 편리하게 MySQL을 운영하기 위해 사용합니다.
* **특징:** 스토어드 프로시저 역시 테이블처럼 데이터베이스 내부에 저장되는 **데이터베이스 개체** 중 하나입니다.

### 1-2. 스토어드 프로시저의 기본 문법
스토어드 프로시저를 묶어주기 위해 구문 형식을 정확히 지켜야 합니다.

* **구분자 변경 (`DELIMITER`):** 
  * 프로시저 안에는 여러 개의 SQL 문이 들어가며 각각 세미콜론(`;`)으로 끝납니다. 이 세미콜론이 프로시저의 끝인지 SQL 문의 끝인지 MySQL이 헷갈리지 않게 하려면, 구분자를 일시적으로 `$$` 등으로 바꾸어 주어야 합니다,.
  * 구문이 끝난 후에는 다시 `DELIMITER ;`를 통해 원래의 세미콜론으로 되돌려 놓습니다.
* **생성 (`CREATE PROCEDURE`):**
  * `CREATE PROCEDURE 프로시저_이름(매개변수)` 형식으로 이름을 지정합니다.
  * 실제 실행할 SQL 프로그래밍 코드는 `BEGIN`과 `END $$` 사이에 작성합니다,.
* **실행 (`CALL`):** 
  * 스토어드 프로시저를 생성한 것은 커피 자판기를 만든 것과 같습니다. 커피를 뽑으려면 실행을 해야 하며, 이때 `CALL 프로시저_이름();` 구문을 사용합니다,.
* **삭제 (`DROP PROCEDURE`):** 
  * 생성할 때는 이름 뒤에 괄호`()`를 붙이지만, 삭제할 때는 `DROP PROCEDURE 프로시저_이름;` 처럼 괄호를 붙이지 않는 점에 주의해야 합니다.

### 1-3. 매개변수의 활용
자판기에 동전을 넣고(입력) 커피가 나오듯(출력), 스토어드 프로시저도 매개변수를 통해 값을 주고받을 수 있습니다.

* **입력 매개변수 (`IN`):** 
  * 프로시저를 호출할 때 값을 전달받기 위해 사용합니다,.
  * *형식:* `IN 매개변수_이름 데이터_형식`.
* **출력 매개변수 (`OUT`):** 
  * 프로시저 안에서 계산된 결과를 호출한 쪽으로 돌려주기 위해 사용합니다,.
  * 프로시저 내부에서는 `SELECT ~ INTO` 문을 사용하여 출력 매개변수에 값을 대입하며, 프로시저를 호출(`CALL`)할 때는 값을 받을 변수를 `@변수명` 형태로 넘겨줍니다,,.

### 1-4. 내부에서의 SQL 프로그래밍 활용
이전에 배운 제어문들을 `BEGIN ~ END` 사이에 넣어 실제 프로그래밍을 구현할 수 있습니다,.
* **변수 선언과 대입:** `DECLARE`를 통해 변수를 준비하고, `SET`을 사용해 값을 대입합니다,.
* **제어문:** 참/거짓에 따라 실행을 나누는 `IF ~ ELSE` 문이나 특정 작업을 반복하는 `WHILE` 문 등을 활용하여 논리적인 처리가 가능합니다,.

### 1-5. 동적 SQL
* **개념:** 테이블 이름이나 조건 등이 고정되지 않고, 프로그램 실행 중에 **다이내믹하게 SQL 쿼리문이 생성 및 변경되어 실행**되도록 하는 기능입니다,.
* **활용:** 
  1. `PREPARE`: 쿼리문을 미리 준비해 둡니다,.
  2. `EXECUTE`: 준비된 쿼리문을 실행합니다,.
  3. `DEALLOCATE PREPARE`: 사용이 끝난 쿼리문을 해제합니다.

## 2. 스토어드 함수와 커서 

### 2-1. 사용자가 직접 만드는 '스토어드 함수'
* **개념:** MySQL이 기본적으로 제공하는 내장 함수(SUM, CAST 등) 외에, 사용자가 필요에 의해 직접 만들어 사용하는 함수입니다.
* **스토어드 프로시저와의 핵심 차이점:** 모양은 비슷하지만 용도와 사용법이 다릅니다.
  * **반환값 필수:** 스토어드 함수는 `RETURNS` 예약어로 반환할 데이터의 형식을 지정하고, 본문 안에서는 `RETURN` 문으로 **반드시 하나의 값을 반환**해야 합니다.
  * **매개변수:** 스토어드 프로시저와 달리 함수는 모든 매개변수가 입력 매개변수이므로, 매개변수 앞에 `IN`을 붙이지 않습니다.
  * **호출 방법:** 스토어드 프로시저는 `CALL`로 실행하지만, 스토어드 함수는 **`SELECT` 문 안에서 호출**하여 사용합니다.
* **함수 생성 권한 설정:** 스토어드 함수를 생성하려면 먼저 MySQL에서 함수 생성을 허용해 주어야 합니다. `SET GLOBAL log_bin_trust_function_creators = 1;` 명령을 한 번 실행해주면 됩니다.

### 2-2. 한 행씩 순서대로 처리하는 '커서'
* **개념:** 테이블에서 데이터를 **한 행씩 차례대로 접근해서 값을 처리**하기 위한 프로그래밍 방식입니다. 주로 스토어드 프로시저 내부에서 사용됩니다.
* **특징:** 첫 번째 행부터 시작하여 마지막 행까지 순차적으로 이동하며 작업을 수행합니다.

### 2-3. 커서의 작동 순서 (5단계)
커서는 일반적으로 다음의 정해진 5단계 순서를 거쳐 작동합니다.

1. **커서 선언하기 (`DECLARE CURSOR`):** 
   * 처리할 데이터를 `SELECT` 문으로 정의하고 커서의 이름을 지정합니다.
2. **반복 조건 선언하기 (`DECLARE CONTINUE HANDLER`):** 
   * 더 이상 읽을 행이 없을 때(마지막 행에 도달했을 때) 어떻게 작동할지 지정합니다. 주로 `FOR NOT FOUND SET endOfRow = TRUE;` 형태로 끝을 판별할 변수 상태를 변경하도록 설정합니다.
3. **커서 열기 (`OPEN`):** 
   * 선언한 커서를 열어 데이터 처리를 준비합니다.
4. **데이터 가져오기 및 처리 (`FETCH` 및 루프):** 
   * `LOOP ~ END LOOP` 반복문 안에서 `FETCH` 문을 사용해 한 행씩 데이터를 읽어옵니다. 
   * 읽어온 데이터로 계산 등을 처리하다가, 2번에서 설정한 `endOfRow`가 `TRUE`가 되면 `LEAVE` 문을 통해 반복문을 빠져나갑니다.
5. **커서 닫기 (`CLOSE`):** 
   * 모든 작업이 끝나면 커서를 닫아줍니다.

## 3. 자동 실행되는 트리거 

### 3-1. 트리거의 기본 개념
* **정의:** 테이블에 삽입(`INSERT`), 수정(`UPDATE`), 삭제(`DELETE`) 등의 DML(데이터 조작어) 작업이 발생할 때 **자동으로 실행되는 프로그래밍 코드**입니다.
* **비유:** 총의 '방아쇠'를 당기면 자동으로 총알이 발사되듯이, 테이블에 지정된 이벤트가 일어나면 자동으로 코드가 실행된다고 하여 트리거라는 이름이 붙었습니다.
* **주요 목적:** 사용자가 데이터를 삭제하거나 수정할 때, 이전 데이터를 백업해두는 등 추가 작업을 잊어버리는 실수를 방지하여 **데이터의 무결성**을 유지하는 데 사용됩니다.

### 3-2. 스토어드 프로시저와의 핵심 차이점
트리거는 스토어드 프로시저와 문법이 비슷하지만 작동 방식에는 큰 차이가 있습니다.
* **직접 실행 불가:** 프로시저처럼 `CALL` 문을 사용하여 사용자가 직접 실행시킬 수 없습니다. 오직 설정된 이벤트(INSERT, UPDATE, DELETE)가 발생할 때만 자동으로 실행됩니다.
* **매개변수 사용 불가:** 프로시저와 달리 외부에서 값을 받거나 돌려주는 `IN`, `OUT` 매개변수를 사용할 수 없습니다.

### 3-3. 트리거가 사용하는 임시 테이블: `NEW`와 `OLD`
트리거가 작동할 때 MySQL 내부적으로 데이터를 임시 보관하기 위해 두 가지 가상의 테이블을 사용합니다. 백업 등의 작업을 할 때 아주 중요하게 활용됩니다.

* **`NEW` 테이블:** 
  * 새로운 값이 들어올 때 잠시 머무는 곳입니다.
  * `INSERT`(새 값)와 `UPDATE`(변경될 새 값) 작업 시에 작동합니다.
* **`OLD` 테이블:**
  * 예전 값이 삭제되거나 변경되기 전에 잠시 저장되는 곳입니다.
  * `DELETE`(삭제될 예전 값)와 `UPDATE`(변경되기 전의 예전 값) 작업 시에 작동합니다.
  * **활용:** 이 `OLD` 테이블에 있는 데이터를 백업 테이블에 `INSERT` 하도록 코드를 짜면, 원본 데이터가 삭제되더라도 안전하게 기록을 남길 수 있습니다.

### 3-4. 트리거의 활용과 주의사항
* **데이터 백업:** 은행 창구에서 계좌를 폐기할 때 해당 정보를 따로 보관하듯이, 실무에서는 회원 탈퇴나 정보 변경 시 트리거를 통해 백업 테이블(Backup Table)에 변경 시간, 변경한 사용자, 변경 전 데이터를 자동으로 기록하도록 설정합니다.
* **⚠️ `TRUNCATE TABLE` 주의점:** 
  * `DELETE` 문으로 데이터를 지울 때는 트리거가 정상적으로 작동합니다.
  * 하지만 **`TRUNCATE TABLE` 문으로 모든 데이터를 한 번에 지울 때는 `DELETE` 트리거가 작동하지 않습니다.** 대량의 데이터를 지울 때 백업이 필요하다면 반드시 `DELETE` 문을 사용해야 합니다.

## 1. 데이터베이스 구축

아래 코드를 MySQL Workbench에 붙여넣은 후,  
**전체 드래그 → 실행 (Ctrl + Shift + Enter)** 하여 데이터베이스를 구축하세요.

```sql
CREATE DATABASE IF NOT EXISTS week6_db;
USE week6_db;

-- 초기화
DROP TABLE IF EXISTS transactions;
DROP TABLE IF EXISTS accounts;

-- 계좌 테이블
CREATE TABLE accounts (
    acc_id INT PRIMARY KEY,
    name VARCHAR(20) NOT NULL,
    balance INT NOT NULL DEFAULT 0 CHECK (balance >= 0)
);

-- 거래 테이블
CREATE TABLE transactions (
    tx_id INT PRIMARY KEY,
    acc_id INT NOT NULL,
    amount INT NOT NULL,
    tx_date DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- 샘플 데이터
INSERT INTO accounts VALUES
(1, '진아', 20000),
(2, '혜인', 8000),
(3, '규서', 55000),
(4, '규영', 12000);
```
