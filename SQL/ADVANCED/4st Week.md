## 2. 제약조건으로 테이블을 견고하게 

> 제약조건(Constraint) 핵심 정리

### 2-1. 제약조건의 개념과 목적
* **데이터의 무결성 :** 데이터에 '결함이 없는 상태'를 뜻합니다. 
* **역할 :** 중복된 이메일이나 가입되지 않은 아이디로 구매하는 등 데이터의 결함이 발생하는 것을 방지하여 데이터의 무결성을 지키는 규칙

### 2-2. 6가지 핵심 제약조건

### ① 기본 키 제약조건 (PRIMARY KEY)
* **개념 :** 테이블의 많은 행 데이터 중에서 데이터를 구분할 수 있는 고유한 식별자 역할을 합니다 (예: 학번, 사번, 주민등록번호 등).
* **특징** 
  * 입력되는 값은 절대 **중복될 수 없으며, 비어있어도(NULL) 안 됩니다**.
  * 테이블당 **단 1개**만 설정할 수 있습니다.
  * 기본 키로 지정된 열은 자동으로 클러스터형 인덱스(찾아보기)가 생성됩니다.

### ② 외래 키 제약조건 (FOREIGN KEY)
* **개념 :** 두 테이블 사이의 관계를 연결해주고 데이터의 무결성을 보장해주는 역할입니다.
* **구조 :** 보통 회원 테이블의 기본 키(기준 테이블)와 구매 테이블의 외래 키(참조 테이블)가 연결되어 일대다 관계를 맺습니다.
* **CASCADE 옵션 :** 기준 테이블의 데이터가 변경/삭제될 때 참조 테이블의 데이터도 함께 자동으로 변경/삭제되도록 해주는 `ON UPDATE CASCADE`와 `ON DELETE CASCADE` 구문을 유용하게 사용할 수 있습니다.

### ③ 고유 키 제약조건 (UNIQUE)
* **개념 :** 기본 키와 거의 비슷하게 '중복되지 않는 유일한 값'을 입력해야 하는 조건입니다.
* **차이점 :** 기본 키와 달리 **NULL(빈 값)을 허용**하며, 하나의 테이블에 **여러 개를 설정**할 수 있습니다. (예: 회원 테이블에서 기본 키는 아이디로 하되, 이메일 주소도 중복되지 않아야 하므로 이메일 열에 고유 키를 설정).

### ④ 체크 제약조건 (CHECK)
* **개념 :** 테이블에 입력되는 데이터를 사전에 점검하는 기능입니다.
* **활용 :** 키(height)에 마이너스 값이 들어오지 않도록 '100 이상만 가능'하게 하거나, 전화번호 국번을 '02', '031' 등 특정 값만 들어오도록 값의 범위를 제한할 때 사용합니다,. 

### ⑤ 기본값 정의 (DEFAULT)
* **개념:** 데이터를 입력할 때 실수로 값을 생략하더라도, 자동으로 입력될 값을 미리 지정해 놓는 방법입니다.
* **활용:** 연락처 국번을 입력하지 않으면 기본으로 '02'가 들어가도록 설정할 수 있습니다.

### ⑥ 널 값 허용 (NULL / NOT NULL)
* **개념:** 데이터에 빈 값을 허용할지 말지 결정합니다. `NULL`은 '아무것도 없다'는 뜻이며 공백이나 0과는 다릅니다.
* **특징:** `NOT NULL`을 설정하면 반드시 값을 입력해야 합니다. `PRIMARY KEY`가 설정된 열은 자동으로 `NOT NULL`로 인식됩니다.

## 3. 가상의 테이블: 뷰 

### 3-1. 뷰(View)의 기본 개념
* **정의 :** 일반 사용자의 입장에서는 테이블과 똑같이 보이지만, 실제로는 데이터를 직접 가지고 있지 않은 `가상의 테이블`을 의미합니다.
* **비유 :** 윈도우 바탕 화면의 `바로 가기 아이콘`과 아주 비슷합니다.
* **실체 :** 뷰의 진짜 실체는 데이터를 추출하는 **`SELECT` 문**입니다. 사용자가 뷰를 조회하면, 내부에 저장된 `SELECT` 문이 실행되어 원본 테이블에서 데이터를 가져와 화면에 보여주는 방식으로 작동합니다.
* **이름 관례 :** 일반 테이블과 뷰를 쉽게 구분할 수 있도록 보통 뷰의 이름 앞에는 `v_`를 붙여서 생성합니다.

### 3-2. 뷰를 사용하는 핵심 이유 (장점)

### ① 보안(Security)에 도움
  회원 테이블에 연락처나 평균 키 등 민감한 개인정보가 포함되어 있을 때, 해당 열을 제외하고 아이디와 이름만 보이는 뷰를 생성해 아르바이트생 등에게 제공할 수 있습니다. 이를 통해 중요한 데이터가 노출되는 것을 사전에 방지할 수 있습니다.

### ② 복잡한 SQL → 단순화
  여러 개의 테이블을 엮는 `INNER JOIN` 등 길고 복잡한 쿼리문을 자주 사용해야 한다면, 이를 뷰로 한 번 만들어 두면 됩니다. 그 이후부터는 복잡한 쿼리 대신 `SELECT * FROM 뷰_이름`처럼 아주 단순한 SQL문 만으로 원하는 결과를 얻을 수 있습니다.

### 3-3. 뷰의 생성, 수정, 삭제
* **생성 (`CREATE VIEW`) :** 
  `CREATE VIEW 뷰_이름 AS SELECT 문` 형식을 사용합니다. 
  또한 `CREATE OR REPLACE VIEW`를 사용하면 기존에 같은 이름의 뷰가 있더라도 오류 없이 새로운 내용으로 덮어쓰기가 되어 편리합니다.
* **수정 (`ALTER VIEW`) :** 이미 만들어진 뷰의 내용을 수정할 때는 `ALTER VIEW` 구문을 사용합니다.
* **삭제 (`DROP VIEW`) :** 필요 없는 뷰를 지울 때는 `DROP VIEW` 구문을 사용합니다.

### 3-4. 뷰를 통한 데이터 수정 및 관리 시 주의점
> 뷰를 통해서 원본 테이블의 데이터를 수정(`UPDATE`)하거나 삽입(`INSERT`)할 수도 있지만, 여러 **제약 조건이 따릅니다**.
* **데이터 입력 불가 조건 :** 만약 원본 테이블에 `NOT NULL`로 설정된 필수 입력 열이 있는데, 뷰에 그 열이 포함되어 있지 않다면 뷰를 통해서는 새로운 데이터를 삽입할 수 없습니다.
* **`WITH CHECK OPTION` :** 특정 조건(예: 키 167 이상)으로 만들어진 뷰에 그 조건을 벗어나는 엉뚱한 데이터(예: 키 159)가 입력되는 것을 방지하려면 뷰를 생성하거나 수정할 때 `WITH CHECK OPTION` 예약어를 사용해야 합니다.
* **원본 테이블 삭제 시 :** 뷰가 참조하고 있는 진짜 원본 테이블을 삭제해버리면, 당연히 뷰도 조회할 수 없게 됩니다 (`CHECK TABLE` 구문으로 뷰의 훼손 상태를 확인할 수 있습니다).

### 3-5. 단순 뷰와 복합 뷰
* **단순 뷰 :** 1개의 테이블만 사용하여 만든 기본적인 뷰입니다.
* **복합 뷰 :** 2개 이상의 테이블을 조인(`JOIN`)하여 만든 뷰입니다. 복합 뷰는 원칙적으로 데이터를 입력, 수정, 삭제할 수 없으며 **읽기 전용**으로 주로 사용됩니다.
