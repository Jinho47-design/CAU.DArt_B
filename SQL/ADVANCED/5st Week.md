# 1️⃣ 학습 내용 정리

## 1. 인덱스 개념을 파악하자 

### 1-1. 인덱스(Index)의 기본 개념
* **정의 :** 인덱스는 데이터베이스에서 데이터를 빠르게 찾을 수 있도록 도와주는 도구입니다. 실무에서는 데이터의 양이 방대하기 때문에 인덱스 없이 데이터베이스를 운영하는 것은 현실적으로 불가능합니다.
* **비유 :** 두꺼운 책의 맨 뒤에 있는 '찾아보기(색인)'와 아주 똑같은 개념입니다. 찾아보기가 있다면 원하는 단어가 있는 페이지를 바로 찾을 수 있듯이, 인덱스가 있으면 전체 데이터를 다 찾아보지 않고도 원하는 결과를 빠르게 얻을 수 있습니다.

### 1-2. 인덱스의 장점과 단점

* **장점** 
  * `SELECT` 문을 사용한 데이터 검색 속도가 매우 빨라집니다.
  * 검색이 빨라지면 컴퓨터의 처리 부담이 줄어들어, 결과적으로 전체 시스템의 성능이 향상됩니다.
* **단점**
  * 찾아보기가 책의 페이지를 차지하듯, 인덱스도 데이터베이스 안에 추가적인 공간을 차지합니다 (대략 테이블 크기의 10% 정도의 공간이 추가로 필요합니다).
  * 처음에 인덱스를 생성하는 데 시간이 오래 걸릴 수 있습니다.
  * **가장 큰 문제점 :** 데이터의 변경 작업(`INSERT`, `UPDATE`, `DELETE`)이 자주 일어나면, 인덱스도 계속 수정해주어야 하므로 오히려 성능이 나빠질 수 있습니다.

### 1-3. 인덱스의 두 가지 종류
MySQL에서 사용되는 인덱스는 크게 **클러스터형 인덱스**와 **보조 인덱스**로 나뉩니다.

### ① 클러스터형 인덱스 
* **특징** 
    * 책의 내용 자체가 알파벳 순서대로 정렬되어 있는 '영어사전'과 같습니다. 
* **생성 방법** 
    * 테이블의 특정 열을 기본 키(`PRIMARY KEY`)로 지정하면, 자동으로 그 열에 클러스터형 인덱스가 생성됩니다.
* **정렬 방식** 
    * 클러스터형 인덱스가 생성되면, 해당 열을 기준으로 **실제 행 데이터들이 자동으로 오름차순 정렬**됩니다.
* **개수 제한** 
    * 실제 데이터를 정렬하는 방식이므로, 테이블당 **단 1개**만 만들 수 있습니다.

### ② 보조 인덱스 
* **특징** 
    * 책의 본문 내용은 그대로 두고, 맨 뒤에 단어별 페이지를 기록해 둔 일반적인 책의 '찾아보기'와 같습니다.
* **생성 방법** 
    * 테이블의 특정 열을 고유 키(`UNIQUE`)로 지정하면, 자동으로 그 열에 보조 인덱스가 생성됩니다.
* **정렬 방식** 
    * 클러스터형 인덱스와 달리, 보조 인덱스를 생성해도 **실제 행 데이터의 순서는 변경되지 않고 그대로 유지**됩니다.
* **개수 제한** 
    * 책 한 권에 찾아보기를 여러 개(예: 인명 색인, 지명 색인 등) 만들 수 있듯이, 하나의 테이블에 여러 개를 생성할 수 있습니다.

## 2. 인덱스의 내부 작동 

### 2-1. 인덱스의 핵심 구조 : 균형 트리 
* **개념** 
    * 클러스터형 인덱스와 보조 인덱스는 모두 내부적으로 **균형 트리** 구조로 만들어집니다. 이는 데이터를 검색하는 데 아주 뛰어난 성능을 발휘하는 범용적인 자료 구조입니다.
* **구조** 
    * 나무를 거꾸로 뒤집어 놓은 형태를 가지며, 가장 최상단의 뿌리를 **루트 노드(Root Node)**, 줄기를 **중간 노드**, 가장 끝에 달린 잎을 리프 노드(Leaf Node)라고 부릅니다.
* **페이지(Page)** 
    * MySQL에서는 데이터가 저장되는 공간인 '노드'를 페이지(Page)라고 부릅니다. 페이지는 최소한의 저장 단위로 크기는 16Kbyte(16384byte)이며, 데이터를 1건만 입력해도 1개의 페이지가 필요합니다.

### 2-2. 인덱스의 데이터 검색 방식
* **전체 테이블 검색** 
    * 인덱스가 없는 테이블에서 데이터를 찾을 때, 데이터를 처음부터 끝까지 모두 읽어보는 방식입니다.
* **인덱스 검색** 
    * 인덱스가 있을 때 루트 페이지를 시작으로 중간 페이지를 거쳐 리프 페이지를 읽어 데이터를 찾는 방식입니다. 전체 데이터 페이지를 읽는 것보다 속도가 훨씬 빠릅니다.

### 2-3. 인덱스 단점의 핵심 원인 : 페이지 분할
인덱스를 구성하면 데이터 검색(`SELECT`) 속도는 향상되지만, 데이터의 변경 작업(`INSERT`, `UPDATE`, `DELETE`) 시에는 성능이 나빠집니다.
* **페이지 분할** 
    * 데이터를 새로 삽입(`INSERT`)할 때 해당 페이지에 빈 공간이 없다면, 새로운 페이지를 준비해서 데이터를 나누는 작업이 발생합니다.
* **성능 저하** 
    * 페이지 분할이 일어나면 상위 노드(루트 노드 등)까지 새롭게 구성해야 하므로 작업 시간이 오래 걸리고 MySQL이 느려집니다. 이것이 인덱스가 있을 때 데이터 변경 작업이 느려지는 근본적인 이유입니다.

### 2-4. 클러스터형 인덱스와 보조 인덱스의 내부 구조 차이
두 인덱스는 구조적으로 큰 차이를 보이며, 이로 인해 검색 방식과 속도가 다릅니다.

* **클러스터형 인덱스의 구조**
  * 기본 키(Primary Key)를 지정하면 생성되며, 지정된 열을 기준으로 **실제 데이터 페이지들이 정렬**됩니다.
  * 가장 큰 특징은 **인덱스의 '리프 페이지'가 곧 '실제 데이터 페이지' 자체**라는 점입니다.
  * 루트 페이지에서 리프 페이지로 이동하면 바로 원하는 데이터를 얻을 수 있습니다.

* **보조 인덱스의 구조**
  * 고유 키(Unique Key)를 지정하거나 직접 인덱스를 생성할 때 만들어지며, **실제 데이터 페이지의 순서는 변경되지 않습니다**.
  * 보조 인덱스는 데이터 페이지와 별개로 독립적인 인덱스 페이지(루트, 리프)를 구성합니다.
  * 보조 인덱스의 리프 페이지에는 실제 데이터가 아니라, 데이터가 위치한 '데이터 페이지 번호 + 위치'가 기록되어 있습니다.
  * 데이터를 검색할 때 인덱스의 리프 페이지를 찾은 후, 그곳에 적힌 주소를 보고 **실제 데이터 페이지로 한 번 더 이동**해야 하므로 클러스터형 인덱스보다는 검색 속도가 조금 더 느립니다.

> **확인문제: 다음 설명에서 빈칸에 공통으로 들어갈 용어를 쓰시오.**

```
인덱스를 구성하게 되면 데이터의 변경 작업(INSERT, UPDATE, DELETE)시에 성능이 나빠지는 단점이 있습니다.

(            ) 작업이 일어나면 MySQL이 느려지고 너무 자주 일어나면 성능에 큰 영향을 줍니다.
```
